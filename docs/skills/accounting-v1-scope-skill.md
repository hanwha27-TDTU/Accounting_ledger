# Accounting Ledger V1 Scope Skill

이 문서는 회계장부 앱의 1차 버전 구현 범위를 다룰 때 적용하는 스킬 문서다. 기능 추가, 우선순위 판단, 범위 축소/확대, 2차 보류 여부, 세무사 전달 패키지, 모바일/APK 대비 범위를 결정할 때 이 문서를 먼저 본다.

## 1차 제품 원칙

| 항목 | 결정 |
|---|---|
| 앱 형태 | 단일 HTML + GitHub Pages |
| 모바일 | PWA 가능 + Capacitor-ready 설계 |
| 사용자 | owner 1명 중심 |
| 데이터 | IndexedDB/localStorage + Supabase + Cloudinary |
| 회계 원장 | 내부 복식부기 SSOT |
| 입력 UX | 간편장부식 쉬운 입력 |
| AI | 화면 노출 없음, future 필드만 |

## 앱 0.02 구현 기준선

| 구분 | 현재 상태 |
|---|---|
| 구현 | 업무 대시보드, 사업자 설정, 2025 소급 거래 입력, 자동 복식분개, 장부·전표 검토, IndexedDB, JSON 백업·복원, 운영 Supabase 공개 연결·익명 RLS·Google provider 사전 진단, 개발자 레지스트리 |
| 부분 구현 | VAT 후보 분리, 증빙 누락, 신고 준비도, 마감 준비 상태, 일반 동기화·canonical 변경 감지 구조 |
| 미구현 | Google provider 설정과 실제 owner OAuth·인증 RLS 왕복, Cloudinary 파일 업로드, Excel import/export, 법정서식 출력, 세무사 전달 패키지, 최종본 지정 UI |

0.02 기준선이 생겼다고 V1 전체가 완료된 것으로 표시하지 않는다. 이후 기능은 완료 조건을 통과할 때마다 앱 버전을 `0.01`씩 올린다.

## 구현 판단 규칙

1. 1차 범위에 없는 기능은 구조와 메뉴 자리만 만들 수 있지만, 완료 기능처럼 표시하지 않는다.
2. 보류 기능을 구현해야 할 것처럼 보이는 경우 먼저 이 문서와 메인 설계서의 2차 보류 목록을 확인한다.
3. 회계 코어, 저장/동기화, 증빙 보관, 백업, 리포트 최신서식 검증은 1차에서 빠지면 안 된다.
4. 세무사 전달은 파일 패키지 내보내기만 제공하고, 로그인 공유는 만들지 않는다.
5. AI는 API 키가 없으므로 사용자 화면에 노출하지 않는다.
6. 모바일을 염두에 두되 APK 빌드는 1차 완료 조건이 아니다.
